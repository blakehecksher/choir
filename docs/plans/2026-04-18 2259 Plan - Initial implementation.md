# Plan - Initial implementation
_Created: 2026-04-18 2259_

## Goal

Ship all five phases of the Choir Visualizer spec in one implementation pass: static choir, audio loading, mouth reactivity, per-mouth variation, conductor dots. Runnable as static files on GitHub Pages.

## Approach

Flat file layout per spec §2.7: `index.html`, `style.css`, `main.js`, `songs.json`, `assets/`. No build step, no framework.

### Phase 1 — Static choir
- `index.html` with inline SVG viewBox 1000x600
- Three rows of ellipses generated by `main.js` on DOMContentLoaded (9/10/10, horizontally staggered)
- Beige background `#E8CFA0`, black ellipses (rx=14, ry=1 default so they read as dashes)
- Corner icons (shuffle + upload) bottom-right, low contrast
- Single white conductor dot centered below choir, CSS breathing animation (4s, scale 0.9↔1.1, opacity 0.7↔1.0)

### Phase 2 — Audio loading
- `songs.json` with 5 curated entries (title, composer, url, bpm)
- On load: fetch songs.json, pick random
- Click dot → resume AudioContext, connect `<audio>` → MediaElementSource → AnalyserNode (fftSize 2048) → destination, start playback
- Shuffle button: fade out 300ms, pick new random entry, return to IDLE
- Upload button: file picker → ObjectURL → load as current song, bpm default 80
- On CORS/load failure: log, skip to next song silently

### Phase 3 — Mouth reactivity
- Per-frame `getByteFrequencyData()`
- Compute amplitude for 3 bands (soprano 300–1100, alto/tenor 200–500, bass 80–250) by averaging FFT bins in range, normalize 0–1
- Drive each mouth's `ry` from its row's band amplitude (synced within row)

### Phase 4 — Per-mouth variation and easing
- Each row keeps ~30-frame ring buffer of recent amplitudes
- Each mouth has stable `lagFrames` (0–9) and stable noise phases
- Per-mouth target = bandAmpLagged × (1 + noise × 0.3)
- Easing: exponential approach with ~80ms attack / ~200ms release time constants
- Mouth `rx` shrinks slightly at max openness (14 → 12) so open mouths read as round, not wide

### Phase 5 — Conductor dots
- SVG path: figure-eight lemniscate centered below choir
- IDLE: single dot pulsing (CSS)
- Click → 800ms split animation (elastic), hide single dot, show two dots
- PLAYING: two dots animate along path via `getPointAtLength`, 180° phase offset, 4 beats per cycle, per-beat ease-in-out so velocity varies naturally
- Song ends → ENDING state (2s): settle mouths closed, merge dots (1.2s) → IDLE

## Scope boundaries

Explicitly out per spec §5:
- Mobile-specific tuning
- Accessibility beyond basic alt text
- Pause/resume (optional; skipping)
- Progress bar, time display, track info UI
- Settings, playlists, sharing
- Stem separation
- Streaming service integration

## Open questions

- **CORS-verified song URLs.** Can't test CORS from static scaffolding. Will include 5 placeholder entries with Internet Archive URLs; note in `state.md` that user may need to verify/replace. App handles missing audio by skipping silently. Uploads work regardless.
- **Favicon.** Skipping for initial version (not required by spec); `assets/` stays empty.
